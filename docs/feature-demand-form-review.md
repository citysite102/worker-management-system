# 需求單欄位檢視與設計決策（grilling 修正版）

> 狀態：釐清中（2026-07-24）｜這是寫正式規格前的「欄位檢視＋地基決策」，不是最終規格。
> 來源：業主提供的 CSV「雇主的需求單 - 需求單」（欄位＋逐角色權限矩陣）。
> 依據：`docs/marketplace-platform-spec.md` §11、`docs/marketplace-lead-pipeline.md`。

## 0. 一句話定位（grilling 定案）

**這整份「需求單」是在擴充*內部案件模組*（客戶委託仲介找人），不是做公開媒合；公開站只是進件漏斗。** 先前「以 `jobPostings` 為主」的判斷已作廢，正確載體是**內部 `caseDemands` ＋ 案件**。

## 1. 地基決策（經 grilling 釘死）

| #   | 決策                                                                                          | 理由                                                                                                               |
| --- | --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| D1  | **連連看＝後台單向指派、免逐筆同意**                                                          | 仲介與移工有委任關係；逐筆同意不符實際作業。用既有 `caseAssignment`。                                              |
| D2  | **候選人池＝內部名冊 `workers`**                                                              | 內部移工無登入帳號，只透過案件鏈連到工作。                                                                         |
| D3  | **自助移工進池＝註冊 → 開「允許仲介主動配對」opt-in → 客服勾稽收編成 `workers` → 走既有指派** | 沿用既有 `workerPublicProfiles.workerId` 勾稽機制；opt-in 為新增旗標。                                             |
| D4  | **同意模型：內部靠委任；自助靠「一次性常設 opt-in」，非逐筆**                                 | 一次開權限，之後主動配對免再問。                                                                                   |
| D5  | **需求單載體＝內部 `caseDemands`，不合併寫入表**                                              | `jobPostings`（雇主自助）與 `caseDemands`（內部）擁有者/生命週期不同。                                             |
| D6  | **（c）「來源統一」彈性＝YAGNI，統一只在讀取層**                                              | 對外顯示已由 `publicView`／`resolveTarget`／`listPublicOpenDemands` 統一；不需合併寫入表。等雇主自助真的上線再長。 |
| D7  | **公開曝光沿用既有 `listPublicOpenDemands`**                                                  | 內部需求單本來就會浮現到公開站；CSV 的「一般大眾/雇主 View」對應這條橋，非雇主自助張貼。                           |

## 2. 欄位檢視（對照*內部案件模型*）

圖例：✅ 已有｜🟡 大致有/需確認｜🆕 新增

### A. 雇主基本資料（← 案件的客戶 `cases.customerId → customers`）

| CSV 欄位     | 內部對應                                                                      | 狀態 | 備註                                                                                                                 |
| ------------ | ----------------------------------------------------------------------------- | ---- | -------------------------------------------------------------------------------------------------------------------- |
| 雇主全名     | `customers.name`                                                              | ✅   | 一般大眾不可 View（去識別）                                                                                          |
| 顯示名稱     | （無）                                                                        | 🆕   | 一般大眾可 View → **匿名代稱**（如「北市・家庭看護」），真名永不外露；此欄需送審 Review。放 `customers` 或 `cases`。 |
| 雇主情況備註 | `customers.notes` / `cases.caseCondition` / `caseQualifications.employerNote` | 🟡   | 僅後台；需確認落在哪一個                                                                                             |

### B. 職缺／需求（← `caseDemands` ＋ `cases`）

> ⚠️ 重點修正：這組「公開職缺欄位」多數只在 `jobPostings`，**對 `caseDemands` 幾乎全是新增**。

| CSV 欄位               | 內部對應                                     | 狀態 | 備註                                                            |
| ---------------------- | -------------------------------------------- | ---- | --------------------------------------------------------------- |
| 職稱                   | `caseDemands.label`                          | 🟡   | `label`(notNull,100) 可能即職稱；需確認語意，或改為自由文字職稱 |
| 職類                   | `caseDemands.qualType`                       | ✅   | 8 類吻合                                                        |
| 工作縣市               | `cases.publicCity`（對外）/ 案件地址（內部） | 🟡   | 縣市在案件；「區(選填)」需新增                                  |
| 需求人數               | `caseDemands.neededCount`                    | ✅   |                                                                 |
| 聘僱型態               | （caseDemands 無）                           | 🆕   | 住家/不住家/機構/其他；enum 與 jobPostings 一致                 |
| 薪資（上下限選填）     | （caseDemands 無）                           | 🆕   | salaryMin/Max                                                   |
| 期望上工日             | （caseDemands 無）                           | 🆕   | 對外                                                            |
| 實際預計上工日         | （無）                                       | 🆕   | 僅後台；「因文件資格有狀況」                                    |
| 條件要求               | （caseDemands 無）                           | 🆕   | 對外                                                            |
| 公開說明               | （caseDemands 無）                           | 🆕   | 對外                                                            |
| 案件情況備註(密)       | `caseDemands.notes` / `cases.caseCondition`  | 🟡   | 僅後台                                                          |
| 案件情況備註(求職者用) | （無）                                       | 🆕   | 後台寫、求職者/應徵者可 View → 走 publicView                    |
| 案件情況備註(應徵者用) | （無）                                       | 🆕   | 後台寫、僅配對到的應徵者 View → 走 publicView                   |

> 這等於把 `caseDemands` 補成「完整需求單」，讓它的對外卡片能跟 `jobPostings` 一樣豐富（現在 `toPublicDemandCard` 因缺欄位而較稀疏）。

### C. 目前應徵者（← 連連看到案件的 `workers`，＋新增欄位）

> 這段對應「被後台連連看配對到此需求的內部移工」。

| CSV 欄位                      | 內部對應                                   | 狀態 | 備註                                                                                |
| ----------------------------- | ------------------------------------------ | ---- | ----------------------------------------------------------------------------------- |
| 求職者全名                    | `workers.name`                             | ✅   | 隱私控管                                                                            |
| 求職者名稱匿名                | `workerPublicProfiles.alias`（僅收編者有） | 🟡   | 純內部移工無 alias → 需新增或僅適用收編者                                           |
| 身高／體重／學歷／宗教        | （`workers` 無）                           | 🆕   | 全部新增於 `workers`                                                                |
| 工作經驗-時間／內容           | `workerExperiences`（僅有帳號者）          | 🟡   | 自填經歷綁登入帳號；純內部移工無 → 需確認經歷資料來源                               |
| 工作經驗-地點／薪資／離職原因 | （`workerExperiences` 無）                 | 🆕   | 新增欄位                                                                            |
| 工作經驗-時任雇主評價         | `ratings`（平台驗證）                      | 🟡   | 自述 vs 驗證評分不可混用；建議 `workerExperiences` 新增「自述、未驗證」欄，明確標示 |

## 3. 權限矩陣如何落地（不是重寫，是翻譯到既有中介層）

CSV 橫向五角色，剛好對應既有機制：

| CSV 角色         | 既有對應                                                  |
| ---------------- | --------------------------------------------------------- |
| 一般大眾         | `publicProcedure` ＋ `publicView` 去識別輸出              |
| 雇主             | `employerProcedure`（「Create 需付費」＝未來付費牆 gate） |
| 後台管理人       | `staffProcedure`（full CRUD ＋ Review）                   |
| 有帳號的求職者   | `workerProcedure`／登入；「符合自己身份的」＝依身份過濾   |
| 被連連看的應徵者 | 配對關係（`caseAssignment`）＋ publicView 受眾分層        |

- 對外可見欄位（顯示名稱、求職者匿名、三種備註中對外那兩種、應徵者摘要）**一律經 `publicView` 蓋章輸出**——「誰能看什麼」由型別/結構強制。
- 「Update（Review）」＝送審流程，沿用既有 moderation（`recordModeration`）。

## 4. 尚待釐清（寫正式規格前）

1. 「顯示名稱」放 `customers` 還是 `cases`？匿名代稱的產生規則與 Review 流程。
2. `caseDemands.label` 是否即「職稱」，或需另立自由文字職稱欄。
3. 三種備註各自獨立欄位、受眾邊界（求職者用 vs 應徵者用）確認。
4. 「工作經驗」資料來源：純內部移工沒有 `workerExperiences`（那綁登入帳號）——內部移工的經歷要新開資料表，還是只在收編者身上顯示？
5. opt-in「允許仲介主動配對」旗標放 `workerPublicProfiles` 還是 `users`；收編（勾稽）的觸發時機與 UI。
6. 付費牆：本期只留 gating seam，金流另案。
7. 分期建議：P1 職缺欄位（含三備註、顯示名稱）→ P2 應徵者/經歷欄位 → P3 連連看顯示與 opt-in 收編流 → P4 付費牆與逐欄 Review。

## 5. 已作廢的先前判斷（避免回頭走錯）

- ❌「以 `jobPostings` 為主」——載體是內部 `caseDemands`。
- ❌「連連看需要全新關聯表」——用既有 `caseAssignment`；自助移工先勾稽收編。
- ❌「需要逐筆同意閘門」——仲介模式免逐筆；自助靠常設 opt-in。
