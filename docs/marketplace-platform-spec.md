## 1. 背景與願景

現有系統是**仲介內部後台**：管理移工、雇主（客戶）、案件、資格、媒合、僱傭與各項證件/健檢/保險期限。本專案將其**往下游衍生**成一個**公開、雙邊、仲介居中**的媒合平台：

- **雇主端**：可自行註冊、張貼「工作需求（＝後台的需求單）」，經審核後上架。
- **移工端**：可自行註冊、完善基本資料、產生簡版履歷（參考 104 / Cake），並呈現**過往僱傭/工作紀錄**（本平台最核心的價值）。
- **公開站**：首頁、找工作（少數工作類型可篩選）、找移工（雇主自行搜尋）。
- **仲介（我方）**：所有需求單第一階段需**我方審核**；雇主與移工**不直接聯繫**，實際媒合一律經我方居中。

### 一句話定位

> 一個讓雇主看得到「移工在本平台的可信工作紀錄」、讓移工用母語呈現履歷、但成交前一切聯繫都由仲介把關的媒合平台。

## 4. 範圍與分期

### 4.1 In-scope（本規格涵蓋）

公開站（首頁/找工作/找移工）、移工與雇主註冊登入、移工履歷與工作紀錄、需求單張貼與審核、仲介居中媒合意向、多語系、審核後台、通知。

### 4.2 Out-of-scope（本期不做，僅預留）

金流/計費（§9 僅設計、後期實作）、雙方即時聊天、行動 App（先 RWD 網頁）、AI 智慧推薦媒合、電子簽約。

### 4.3 分期（詳見 §14）

- **P0 地基**：多角色帳號/認證、權限硬化、i18n 框架、公開/後台分流。
- **P1 雇主側**：需求單張貼 + 審核 + 自動轉 case + 找工作頁（開放匿名瀏覽，聯絡才登入【§15-1 v1.1】）。
- **P2 移工側**：移工註冊、履歷、平台工作紀錄、自填經歷審核、找移工頁。
- **P3 媒合**：媒合意向（仲介居中）、通知、揭露流程。
- **P4 金流**：張貼/曝光/媒合成功收費。

## 6. 使用者旅程

### 6.1 移工（求職者）

1. 用 LINE/FB/Google/Email 或手機(WhatsApp OTP) 註冊 → 選語言（vi/id/en）。
2. 引導式完善基本資料（分步驟：身分、國籍、職類、技能、語言、可上工時間）。
3. 系統帶入**平台既有僱傭紀錄**（若此帳號被客服勾稽到既有 `workers`），移工可**自填補充**海外/其他經歷。
4. 產生**簡版履歷**（匿名對外；自己看得到完整版）。
5. 瀏覽「找工作」→ 篩選（看護/房務/其他）→ 對有興趣職缺按「我有興趣」→ 生成媒合意向送客服（不直接聯繫雇主）。
6. 收到客服通知（WhatsApp/LINE/Email）安排後續。

### 6.2 雇主

1. 註冊（社群/Email）→ 選繁中。
2. 填公司/個人雇主基本資料。
3. **張貼需求單**（職類、地點、需求人數、條件、期望上工日、說明）→ 送審。
4. 需求單狀態：審核中 → 通過上架 / 退件補正。**通過後系統自動在後台建立對應 case**。
5. 通過後可進「找移工」→ 瀏覽**匿名履歷** → 對合適移工送「媒合意向」給客服。
6. 客服居中安排；成交前不取得移工私密聯絡資訊。

### 6.3 審核人員（Staff）

1. 審核佇列：待審需求單 / 待審履歷 / 待審自填經歷 / 待處理媒合意向。
2. 需求單審核：檢查合法性/完整性 → 通過（自動轉 case、指派負責人）或退件（附理由）。
3. 履歷/經歷審核：檢查不實/違規內容。
4. 媒合意向處理：聯繫雙方、必要時揭露聯絡資訊、推進到內部媒合流程。

## 7. 功能規格（分模組）

### 7.1 帳號與認證（P0）

- **登入方式**：LINE / Facebook / Google（OAuth）、Email+密碼、手機/WhatsApp OTP。
- **角色**：註冊時選「我是移工 / 我是雇主」→ 決定 `accountType` 與後續資料表。
- **帳號↔資料表**：
  - 移工帳號 → 建立/連結一筆 `workers`（標 `source=self_registered`, `verifiedByStaff=false`）。
  - 雇主帳號 → 建立/連結一筆 `customers`（同上）。
  - 客服可於後台**勾稽/合併**自助帳號與既有名冊（§15-9 定案）。
- **現況相依**：目前為 Manus SDK 單一 OAuth。多社群 + WhatsApp + Email/密碼需：
  - (a) 確認 Manus SDK 是否支援這些 provider；或
  - (b) 缺口則導入獨立認證方案（如 Auth.js / Clerk / Supabase Auth / 自建 OTP）。（§15-11 定案：先評估 SDK，缺口用獨立認證層）
- **驗證**：Email 驗證信、手機/WhatsApp OTP；雇主可加「營利事業/身分」驗證徽章。

### 7.2 移工端：個人資料、履歷、工作紀錄（P2）

- **私有正本**（沿用 `workers`）：姓名、生日、國籍、職類、證件、健檢、附件——**永不對外公開**。
- **公開履歷**（新 `worker_public_profiles`）：
  - 匿名顯示：代號（如 `TW-IDN-00231`）或暱稱、國籍、年齡區間、職類、技能標籤、語言、可上工時間、自我介紹（多語）。
  - **不含**：真實姓名、證件、電話、精確地址、照片原圖【照片處理見 §15-12】。
- **工作紀錄**：
  - **平台可信紀錄**（來自 `case_employments` / `case_assignment_workers`）：職務、期間、狀態，標「✔ 平台驗證」。
  - **自填經歷**（新 `worker_experiences`）：僱主型態（家庭看護/機構/製造…）、職務、期間、內容描述，標「自述」，**需審核**。
- **履歷輸出**：P2 先做平台履歷頁；PDF 下載後期實作（§15-6 定案）。

### 7.3 雇主端：需求單張貼與審核（P1）

- **張貼表單**（對應並擴充 `case_demands`/`case_qualifications`）：
  - **必填**（§15-5 定案）：職類（看護/房務/其他）、工作地點（縣市/區）、需求人數、聘僱型態。
  - **選填**：期望上工日、條件（語言、經驗、性別偏好等，受法遵限制見 §11）、公開說明、期望薪資（以區間顯示）。
- **狀態機**：`draft → pending_review → approved(上架) / rejected(退件)`；上架後 `paused / filled / closed`。
- **審核通過**：自動建立 `case`（含一筆 `case_qualification` + `case_demand`），指派預設負責人；需求單與 case 雙向關聯（§10）。
- **退件**：附結構化理由 + 補正說明，雇主可修改後重送。

### 7.4 公開站（P1/P2）

- **首頁**：平台介紹、熱門職缺、CTA（我要找工作 / 我要找移工）、多語切換、信任訴求（可信工作紀錄、仲介把關）。
- **找工作**（**開放匿名瀏覽**列表與詳情；卡片/詳情皆去識別，不含雇主 PII【§15-1 v1.1】）：
  - 篩選：職類（看護/房務/其他）、地點、聘僱型態。工作類型少，用大按鈕/分頁呈現。
  - 卡片：職類、地點、簡述（機翻 + 原文）、雇主類型（不揭露雇主身分細節）。
  - 動作：「我有興趣」→ 媒合意向；**點擊時才需登入**（未登入導向登入頁並帶 `?next=`）。
- **找移工**（**開放匿名瀏覽**去識別匿名履歷【§15-1 v1.1】）：
  - 篩選：職類、國籍、語言、可上工時間、經驗年資。
  - 卡片：**匿名履歷摘要** + 平台驗證紀錄徽章。
  - 動作：「送出媒合意向」→ 客服居中；**送出時才需雇主登入且需通過的需求單**（未登入導向登入頁）。

### 7.5 媒合與聯繫（仲介居中，P3）

- 任何一方發起「媒合意向 / 我有興趣」→ 建立 `match_requests`（狀態 `new → staff_handling → introduced → matched / closed`）。
- 雙方在平台上**看不到彼此私密聯絡資訊**；由 Staff 判斷後**手動揭露**或線下安排。
- 揭露動作寫入稽核（誰、何時、揭露了什麼）。
- **後半段（點擊之後的業務線索管線）詳見 `docs/marketplace-lead-pipeline.md`**：聯絡偏好捕捉、即時 ack、指派/SLA/觸及狀態、雙邊接續、去重、結案分類與業務工作台（第一階段 lead-gen → 業務手動媒合的核心骨架）。

### 7.6 多語系（P0 框架，全程）

- **介面字串**：訊息目錄（zh-TW / vi / id / en），依使用者選擇或裝置語言。
- **使用者內容**（需求單、履歷、自填經歷、自我介紹）：**機器翻譯**，儲存原文 + 各語言快取，UI 標「機器翻譯，附原文」。
- **翻譯服務**抽象成介面（provider 可換）【§15-11】。
- **結構化欄位**（職類、地點、語言…）用選項對應各語言，免翻譯、最準。

### 7.7 通知（P3）

- 事件：註冊驗證、需求單審核結果、媒合意向進度、客服訊息。
- 管道：Email、WhatsApp/LINE【§15-7】；站內通知中心（沿用現有 NotificationBell 概念，區分公開/後台）。

### 7.8 金流（P4，僅設計）

- **計費模式暫緩**（#8 未定案，留待 P4 再議）：候選為需求單上架費、加值曝光（置頂/精選）、媒合成功費、雇主訂閱方案。
- 資料模型預留：`orders` / `payments` / `plans`；串接金流商（綠界/藍新/Stripe）待定。

## 9. 資料模型變更

### 9.1 既有表擴充

- `users`：新增 `accountType enum('worker','employer','staff')`；`workerId int?`、`customerId int?`（帳號連結）；`preferredLang enum('zh-TW','vi','id','en')`；`phone`、`phoneVerified`。`role` 續用於 staff/admin。
- `workers`：新增 `source enum('staff_created','self_registered')`、`verifiedByStaff bool`、`accountUserId int?`。
- `customers`：同上（`source`、`verifiedByStaff`、`accountUserId`）。
- `case_demands` / `case_qualifications`：新增 `jobPostingId int?`（回連公開需求單）。

### 9.2 新增表

| 表                          | 重點欄位                                                                                                                                                                       | 用途                         |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------- |
| `job_postings`              | employerUserId, customerId, jobType, location, headcount, employmentType, requirements, publicDescription, salaryDisplay, moderationStatus, rejectReason, caseId?, publishedAt | 公開需求單（審核後轉 case）  |
| `worker_public_profiles`    | workerId, alias, headline, skills(json), languages(json), availability, selfIntro, publishStatus, moderationStatus                                                             | 匿名公開履歷                 |
| `worker_experiences`        | workerId, employerType, role, startDate, endDate, description, reviewStatus                                                                                                    | 移工自填經歷（需審核）       |
| `match_requests`            | initiatorType(worker/employer), initiatorUserId, targetType(job/worker), targetId, status, assignedStaffId, note                                                               | 仲介居中媒合橋樑             |
| `moderation_events`         | entityType, entityId, action(approve/reject), reason, staffId, createdAt                                                                                                       | 審核稽核軌跡                 |
| `content_translations`      | entityType, entityId, field, lang, text, source(machine/human)                                                                                                                 | 內容機翻快取                 |
| `audit_logs`                | actorUserId, action, entityType, entityId, meta, ip, createdAt                                                                                                                 | 全站操作稽核（公開平台必備） |
| `orders` / `payments`（P4） | —                                                                                                                                                                              | 金流預留                     |

> 所有新表沿用現有慣例：`int autoincrement` 主鍵、`createdAt/updatedAt` timestamp、外鍵索引、YYYY-MM-DD varchar 存日期。

## 10. 審核工作流（狀態機）

**需求單**

```
draft → pending_review →┬─ approved  → (自動建立 case) → published → paused/filled/closed
                        └─ rejected → (雇主補正) → pending_review
```

**履歷 / 自填經歷**

```
draft → pending_review →┬─ approved → visible
                        └─ rejected(附理由) → draft
```

**媒合意向**

```
new → staff_handling → introduced →┬─ matched
                                   └─ closed(不成/取消)
```

- 每次狀態轉移寫入 `moderation_events` / `audit_logs`。
- 審核 SLA 與退件理由清單【§15-4】。

## 11. 隱私、資安與法遵（重要）

1. **移工 PII 絕不公開**：護照、居留證、身分證、健檢、精確地址、電話——公開層一律不得回傳（後端層級過濾，非只前端隱藏）。匿名履歷只給去識別後的子集。
2. **揭露受控**：真實身分/聯絡資訊只有 Staff 在媒合流程中手動揭露，且全程稽核。
3. **權限硬化（P0 前置，阻斷級）**：現況**所有 procedure 皆 public**，公開上線前必須：
   - 公開 API 與後台 API 分離；
   - 依角色套 `protectedProcedure` / `employerProcedure` / `workerProcedure` / `staffProcedure`；
   - 資源層授權（雇主只能看自己的需求單、移工只能編自己的履歷）。
4. **就業服務法/廣告合規**【§15-15，需你方法務確認】：在台招募/媒合外國人受法規約束，「直接刊登外國人求職廣告」「性別/年齡等條件」可能受限；仲介居中設計有助降低風險，但**條件欄位與文案需法遵審視**。
5. **個資法（PDPA）**：蒐集告知、同意、目的限制、可刪除/停用帳號、資料保留政策【§15-14】。
6. **一般資安**：OTP 節流、暴力破解防護、上傳檔案掃描、Rate limiting、CSRF/XSS、稽核日誌、機密不落前端。

## 16. 風險與假設

- **法遵風險**（最高）：外國人媒合/廣告受就業服務法與相關子法規約束，設計須經法務確認（§15-15）。
- **認證相依**：多 provider + WhatsApp OTP 可能需引入額外認證基建，影響 P0 工期。
- **權限硬化**：現況全 public procedure，公開上線前必須完成，否則有嚴重資料外洩風險。
- **翻譯品質/成本**：機器翻譯需附原文並標示，避免誤譯造成糾紛。
- **假設**：zh-TW 為系統 base 語言；移工/雇主自助帳號預設為未驗證，需客服勾稽；需求單通過即轉 case 並沿用現有後台流程。
